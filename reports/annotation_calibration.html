标注校准页

这个页面帮你在进入人工复核工作台之前统一判断标准：你只需要人工决定 5 个字段，其余 run 元数据和自动标签由系统填充。

报告入口 项目进展 项目全景图 数据质量报告 人工复核工作台

你真正需要标什么

每条样本先看 issue、gold patch、agent patch 和测试结果，再填写下面 5 项。不要改 run id、task id、自动 failure、模型名、路径等系统字段。

patch_quality

判断 agent patch 和 gold patch 的接近程度。

human_failure

人工确认主要失败原因，必要时覆盖自动标签。

sft_usable

这条轨迹是否适合教模型“应该怎么做”。

dpo_usable

这条失败 patch 是否适合和 gold patch 组成偏好对。

notes

写清楚依据：看到了什么，所以这么判。

推荐顺序：先判断是不是环境问题，再看 patch 是否为空，然后比较 agent patch 与 gold patch 的文件、函数、核心逻辑，最后决定 SFT/DPO 是否可用。

OpenAI SWE-bench Verified 给我们的参考

OpenAI SWE-bench Verified 是公开的人类开发者复核样本集。它不直接标 agent 轨迹，但它告诉我们：一个 SWE-bench 任务本身是否清楚、测试是否合理、任务难度如何。

可借鉴的字段
  • underspecified：issue 是否描述不充分，合理开发者是否缺少必要信息。
  • false_negative：测试是否可能拒绝一个合理解法。
  • other_major_issues：任务是否有其他严重质量问题。
  • difficulty：人类开发者预计修复耗时。
怎么迁移到 SWE-Trace
  • issue 清楚、测试合理时，失败更可能归因于 agent patch。
  • issue 明显不清楚时，notes 里写明任务本身有歧义。
  • 测试可能误杀时，谨慎把失败 patch 标成 poor。
  • 难度高不等于 agent 错；它只影响你解释失败原因的语气和置信度。
公开参考来源：OpenAI SWE-bench Verified annotation results 和 SWE-bench Verified 数据集。我们的标注目标更具体：判断 agent 轨迹能否转成 SFT、DPO、debug/reward 数据。

patch_quality 标准

取值 什么时候选 典型判断依据
close agent 改到正确文件和核心逻辑，和 gold patch 思路基本一致，但测试仍未通过或细节有差异。 函数位置相同、条件分支接近、边界处理略缺、命名或小语义不同。
partial agent 找到相关模块或部分原因，但修复不完整，缺少关键分支、测试、兼容逻辑。 改了相关文件但漏掉主要约束；只修一种输入；只加测试不修实现。
poor 方向明显错误，即使 patch 非空，也和 issue/gold patch 的核心修复关系很弱。 改错文件、删除功能、绕过测试、引入无关重构、误解 issue。
empty 没有有效 patch，或 patch 只有日志、格式噪声、无意义注释。 diff 为空；只改 README；只留下临时输出。
env_only 主要失败来自 Docker、依赖、数据集、网络、权限、评测环境，不应该评价 agent patch。 环境没起来、镜像失败、安装失败、dataset load fail、权限错误。

human_failure、SFT 和 DPO 标准

人工判断 建议 原因
环境问题 human_failure 选 EnvironmentError 类；sft_usable=false；dpo_usable=false。 这类样本不能证明 agent 会不会修代码。
失败 run，但 patch 非空且 gold patch 可用 通常 sft_usable=false；如果 agent patch 可作为 rejected，设 dpo_usable=true。 失败轨迹不适合直接模仿，但适合构造偏好对。
已 resolved 或人工确认 patch 正确 可考虑 sft_usable=true；如有可比 rejected，再考虑 dpo_usable=true。 成功轨迹可以教模型按类似方式修复。
patch 方向明显错误 sft_usable=false；只有当错误很清楚且 gold patch 明确时，dpo_usable=true。 DPO 需要“坏答案”和“好答案”的对比清楚。
任务本身不清楚或测试可疑 谨慎纳入；在 notes 写明 issue/test 风险。 这类样本可能把 benchmark 噪声混进训练数据。

校准案例

下面是标注时可直接对照的模式。真实标注时，notes 不用很长，但必须能说明“为什么”。

案例 A：接近 gold patch

patch_quality=close sft_usable=false dpo_usable=true
看什么
agent 改了正确函数，核心 try/branch 位置接近 gold patch，但评测未 resolved。
failure
PatchError.IncompleteFix
notes
Agent touched the correct file and core logic; keep as DPO rejected because the run did not resolve.

案例 B：部分命中

patch_quality=partial sft_usable=false dpo_usable=true
看什么
agent 找到相关模块，但只处理一种输入，gold patch 还有关键兼容分支。
failure
PatchError.IncompleteFix
notes
Patch is in the right area but misses the compatibility branch covered by the gold patch.

案例 C：方向错误

patch_quality=poor sft_usable=false dpo_usable=false
看什么
agent 改了无关文件，或通过删除逻辑绕过问题，和 gold patch 不构成清楚偏好对。
failure
PatchError.WrongBehavior
notes
Patch changes unrelated behavior and does not address the issue described by the task.

案例 D：无有效 patch

patch_quality=empty sft_usable=false dpo_usable=false
看什么
diff 为空，或只有日志、注释、格式噪声。
failure
PatchError.EmptyPatch
notes
No meaningful code patch was produced, so this run is not useful for SFT or DPO.

案例 E：环境失败

patch_quality=env_only sft_usable=false dpo_usable=false
看什么
Docker、依赖、网络、dataset 或权限失败，agent 没有机会完成真实修复。
failure
EnvironmentError.DependencyInstallFail
notes
Run failed before reliable code evaluation because the environment setup did not complete.

案例 F：任务质量可疑

patch_quality=partial sft_usable=false dpo_usable=false
看什么
issue 很不清楚，或测试像 OpenAI SWE-bench Verified 里的 false_negative 风险。
failure
TaskError.Underspecified
notes
Task appears underspecified; avoid using this sample until issue/test quality is reviewed.
