一句话定位

这个项目是一个面向 Code Agent post-training 的数据工程和评测闭环：把 SWE-bench Lite 任务交给 mini-SWE-agent + DeepSeek 真实执行，保存每一步轨迹、patch 和测试结果，再把这些 run 转成 SFT / DPO / Reward 数据，并通过人工复核把自动标签修正成可训练、可解释、可展示的数据资产。

你现在的人工介入点不是写代码修 bug，而是判断 agent 失败样本的质量： patch 离 gold patch 有多近、自动失败标签是否正确、这条样本能不能进入 SFT 或 DPO 数据。

代码模块地图

swetrace/schema

定义 task、trajectory、report、failure label、manual annotation 的稳定数据结构。

swetrace/adapters

把 fake agent 和 mini-SWE-agent 的输出转成统一 artifact。

swetrace/collect

选择 SWE-bench 任务、运行 agent、预拉 Docker 镜像、恢复 orphan raw traj、回填 gold patch。

swetrace/data_builder

从 run artifacts 构建 SFT plan、SFT patch、SFT debug、DPO pairs 和 reward logs。

swetrace/labeling

自动失败分类、人工复核队列、人工 annotation upsert。

下一步人工介入

  1. 打开 /data/yiyuldx/swe/outputs/reports/manual_review_queue.jsonl，选一条 run。
  2. 查看对应 run 目录里的 task.json、gold_patch、patch.diff、report.json 和 test.log。
  3. 判断自动标签是否正确：环境失败、patch 应用失败、未完整修复、测试失败。
  4. 判断 patch_quality：close、partial、poor、empty、env_only。
  5. 优先使用 scripts/serve_review_ui.sh 打开 Web 工作台，或用 scripts/annotate_review.sh 写入人工结论，结果进入 manual_annotations.jsonl。

推荐复核顺序：先看 pydicom 和 pyvista，因为 patch 小、gold patch 明确；再看 astroid、pvlib、marshmallow、sqlfluff。

为什么这个项目现在已经有雏形

它已经不是只有脚本或报告，而是有真实采集、统一 schema、run artifacts、自动标签、训练数据导出、人工复核入口和网页说明。 接下来只要你继续补人工 annotation，就能把“真实失败轨迹”变成可筛选、可解释、可用于后训练实验的数据集。
